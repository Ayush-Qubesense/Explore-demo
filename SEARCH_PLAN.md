# AI Command Palette — Implementation Plan

> ## ✅ BUILT — v0, v1, v2 + the second pass, all verified (40/40 automated checks).
>
> Run it: **double-click `Start Demo.bat`**, then press **Ctrl+K**.
> Rebuild after changing pages, `search` text, or the suggestion examples: `npm run build`
> (= `python build_snapshots.py && node generate_embeddings.js`).
> The build **fails** if a `Try asking` example no longer resolves to its expected page (§10.3).
>
> **§9 and §10 record what implementation changed about this plan** — several assumptions
> below turned out wrong in ways that matter, and §10 documents three real bugs only a
> browser could catch. Read both before trusting §3's details.

Semantic "type what you want, land on the right page" palette for the OpsFlo demo
snapshot site. Fully client-side, no backend, generated by extending
`build_snapshots.py`.

**Scope: the 41-page demo ships first; the production app (QubesenseNextGenOPRSV2) is a
later, separate project.** The demo is nonetheless designed so the port is cheap — several
choices below (§3.2 pre-built `url`s, vanilla JS, the `pages.json` seam) look arbitrary
until you read **§8**, which records what was verified in the real codebase and why it
constrains the demo. Read §8 before changing those choices.

## 1. Architecture (confirmed)

Real sentence embeddings in the browser via **transformers.js**
(`all-MiniLM-L6-v2`, quantized, ~25 MB, WASM). Genuine semantic search, no
backend, no API key, no recurring cost, nothing to fail live on stage.

- The **41 page vectors are precomputed** at build time and shipped as static JSON.
- Only the **user's query** is embedded live (one short string, sub-100 ms warm).
- The model preloads silently on page load, before the user types.
- The embedding call sits behind **one function** so a hosted provider (Azure
  OpenAI / OpenAI) can be swapped in later. Deferred by design.

**Add: a keyword fallback.** If the model fails to load for any reason, the palette
silently degrades to substring matching over the same corpus. Cheap insurance so the
feature can never appear *broken* in front of the CEO — it just appears less clever.

## 2. The single-source-of-truth bridge (replaces the brief's §"generate_embeddings.py")

The brief's Node script can't import Python's `PAGES`. Rather than duplicate the page
model in two languages (which will drift), split generation into two steps:

```
build_snapshots.py   ──emits──▶  assets/data/pages.json     (metadata + embed text, NO vectors)
                                         │
generate_embeddings.js  ──reads──────────┘
                     ──emits──▶  assets/data/page-index.js  (+ 384-float vectors)
                                         │
command-palette.js   ──<script>──────────┘   reads window.__OPSFLO_PAGE_INDEX__
```

**In production this same diagram holds** — only the first box changes (feature docs +
per-user session filter instead of the Python `PAGES` dict). That is the whole point of
the bridge; see §8.3.

- Python stays the **only** place page metadata lives.
- Node does the embedding with the **exact same library and model version** as the
  browser, so corpus and query vectors are bit-for-bit comparable. This is why Node
  beats Python here, and the JSON bridge is what makes Node possible.

Note: `import build_snapshots` from Python would rebuild all 41 pages as a module-level
side effect (the loop at line 592 runs on import). The JSON bridge sidesteps this entirely.

## 3. v0 — semantic palette

### 3.1 New: embedding text design *(this is the quality lever — do not skip)*

Naive `f"{title}. {section}. {' '.join(cols)}"` **fails the brief's own demo queries**:

- "overtime" → Payroll's column is `"OT Hours"`. MiniLM does not reliably know
  `OT` = overtime. The headline demo query misses.
- "who's late" → nothing in `["Vehicle","Job #","ETD","DEP","ETA","ARR","Status"]`
  embeds anywhere near "late".

So add an optional `search` field to `PAGES` entries carrying natural-language
expansions, and build embed text as:

```python
f"{title}. {section}. {', '.join(cols)}. {search}"
```

Mandatory for the 5 archetype pages that have no real `cols`; strongly recommended for
any page whose columns are abbreviations. Seed at minimum:

| Page | `search` text |
|---|---|
| `1676` Payroll | overtime, wages, paychecks, gross pay, hours worked, pay period |
| `1699` Time Card Audit | overtime, clock in clock out, timesheets, hours discrepancy |
| `1728` Deployments / Arrivals | who is late, delayed departures, missed ETD, behind schedule, truck arrivals |
| `1794` Field Tech Tracker | where are my technicians, last seen, tech location |
| `1750` Planner | daily plan, schedule board, dispatch board, jobs for today |
| `1686` Utilization | crew utilization, capacity, how busy are we |
| `1702` Activity Map | live GPS, vehicle locations, map of units |
| `1770` Accounting Dashboard | revenue, financial overview, KPIs, money |
| `1601` Predictive Maintenance | failure prediction, breakdown risk, asset health |
| `1590g/e/c` Certifications | expired certifications, expiring cards, OSHA, compliance |
| `1602` Cert Expiration Tracker | expiring documents, expired paperwork, renewals |
| `1626` Gas Monitors | gas detector, calibration due, H2S monitor |

Budget real time for this table and tune it against the demo query list in §6.
It is the difference between "it understood me" and "it matched a word".

### 3.2 New files

**`assets/data/pages.json`** — emitted by `build_snapshots.py`:
```json
[{ "id": "1676", "title": "Payroll", "section": "Accounting",
   "url": "pages/1676.html", "text": "Payroll. Accounting. Employee, Pay Period, ..." }]
```

> **`url`, not `file` — this is the portability seam (§8).** Each entry carries a
> **ready-made, complete URL**. The palette must never *construct* a URL from an id.
> The demo emits `pages/1676.html`; production emits
> `/Controls/OPRS/1676.aspx?MenuId=1108&SubMenuId=6905&ReportID=`. Same palette code,
> different producer. Building URLs inside the widget is the one shortcut that would
> make the port expensive — see §8.2.

**`generate_embeddings.js`** + `package.json` (sibling to `build_snapshots.py`):
- `npm i @xenova/transformers`
- reads `pages.json`, embeds each `text` (mean-pooled, **normalized**), writes
  **`assets/data/page-index.js`** — *not* `.json` — containing
  `window.__OPSFLO_PAGE_INDEX__ = [ { id, title, section, file, text, vector } ]`.
- Rerun whenever `PAGES` or the `search` text changes.

> **Why a `.js` file, not `.json`.** A `<script>` tag loads under `file://`; `fetch()`
> does not (null origin). Emitting the index as a script assignment costs nothing,
> removes the fetch failure mode entirely, and means a double-clicked `index.html`
> still gets a **working keyword palette** instead of a dead input. Served over http,
> the same file feeds full semantic mode. ~130 KB — irrelevant next to the model.

**`assets/js/command-palette.js`** — the widget:
- read `window.__OPSFLO_PAGE_INDEX__`; preload the pipeline in the background (no spinner).
- debounce ~250 ms → embed query → cosine similarity vs. all 41 (plain JS loop;
  vectors are pre-normalized, so this is just a dot product) → sort → top 5.
- min-similarity threshold to suppress gibberish; keyboard nav (↑/↓/Enter/Esc).
- `Ctrl+K` / `Cmd+K` bound at document level with `preventDefault()`.
- navigate via `pagePrefix + result.file`.
- **fallback:** if the model can't load (`file://`, blocked WASM, anything), substring-match
  `text` instead. The input keeps working and never announces the downgrade.

**`Start Demo.bat`** (repo root) — preserves double-click-to-launch *with* the real model:
```bat
@echo off
cd /d "%~dp0"
start "" http://localhost:8000/index.html
python -m http.server 8000
```
This is what makes decision §6.1 a non-event: the CEO demo is still one double-click,
it just lands on `localhost` instead of `file://`.

**Vendored library** under `assets/frest/assets/vendor/libs/transformers/` — the
`.js`, the WASM binaries, **and** the model files, with `env.allowRemoteModels = false`
and `env.localModelPath` set. Downloading only the JS and letting the model come from
the HF CDN at runtime is the easy mistake here and it defeats the whole purpose.

### 3.3 Modified: `build_snapshots.py`

- **`navbar()` line 225–227** — insert the palette **immediately after** the Legend
  `<li>`, mirroring the real app exactly (see §8.3): `SiteV3.master` has the Legend
  icon at `:117-121` and a Frest **search toggler already sitting at `:124`**, just
  disabled with `d-none`. Match that markup and slot so the port is a copy-paste.
  No `data-page-prefix` is needed — URLs are pre-built in the index (§3.2), which is
  what makes the same markup work in both codebases.
- **`footer_scripts()` line 287** — add three `<script>` tags using `{rel}`, in order:
  the vendored `transformers.min.js`, then `assets/data/page-index.js`, then
  `assets/js/command-palette.js`.
- **New `emit_pages_json()`** — called alongside `launcher()` / `build_landing()`
  at line 641. Build embed text **after** the `setdefault` loop at line 592–594,
  and skip the placeholder `cols` default for archetype pages.
- Rerun `python build_snapshots.py` → then `node generate_embeddings.js`.
  Never hand-edit `pages/*.html`.

> ⚠️ **Brace-escaping footgun.** `head()`, `navbar()`, and `footer_scripts()` are all
> `str.format()` templates. Any literal `{` or `}` in injected CSS/JS **must be doubled**
> (`{{`/`}}`) or the build dies with `KeyError`. This is why `command-palette.js` lives in
> its own file rather than inline — keep injected markup brace-free.

### 3.4 Ranking

1. Embed query with the same model/pooling as the corpus.
2. Dot product vs. all 41 pre-normalized vectors. No permission filter — none exists.
3. Threshold, take top 5.
4. Render `title` + `section` breadcrumb, matching `launcher.html`'s card styling.

## 4. v1 — intent-aware navigation (honestly scoped)

These pages have pre-baked fake data and no backend, so "navigate + server-side filter"
is impossible. The honest version that still lands the payoff:

- small keyword→param rule table in `command-palette.js`
  (`"overdue"` → `?status=Overdue`, `"this week"` → `?range=week`); no LLM needed.
- a small inline script added **once** in `footer_scripts()` (not per-archetype — it

Upgrading to true NL filter extraction later is a one-function swap behind the same
provider-agnostic abstraction as the embedding call.

## 6. Risks & decisions

**Resolved, baked into the plan:**
- *Node vs. Python for embeddings* → **Node**, via the `pages.json` bridge (§2).
- *v1 rules vs. LLM* → **rules**; upgrade only if the audience probes.
- *Serving method* → **`Start Demo.bat` → localhost**; see §6.1.

### 6.1 `file://` — resolved, no architecture change

The question was whether losing `file://` forces a redesign. It does not, and redesigning
around it would be the wrong trade:

- **`file://` is a dead end regardless of the palette.** v2 voice needs a secure context
  (`https://` or `localhost`), which `file://` will never be. Designing the search around
  `file://` would sacrifice the roadmap to save a launcher script.
- **Real embeddings under `file://` aren't feasible anyway.** Chrome blocks module workers
  and `fetch()` from `file://`, and ONNX Runtime needs both to pull its WASM and the ~23 MB
  model. Base64-inlining is fragile and enormous; `--allow-file-access-from-files` is a
  launch flag, i.e. *more* friction than a server, not less.

Two cheap changes make the whole issue a non-event, and they're now in the plan:

1. **`page-index.js` instead of `page-index.json`** (§3.2) — a `<script>` tag loads under
   `file://`, so a double-clicked page still gets a working keyword palette. No dead UI, ever.
2. **`Start Demo.bat`** (§3.2) — double-click is preserved; it just serves `localhost` first.
   Real model, secure context for voice, zero extra steps for the presenter.

Net: **served → full semantic AI. Double-clicked → keyword palette that still works.**
Nothing regresses; the demo path gets strictly better.

**One residual case worth your call:** if the CEO opens the folder *unaided on their own
machine*, `Start Demo.bat` needs Python installed there. If that's the distribution model,
the answer is static hosting (GitHub Pages or any static host — there's no backend, so it
"just works," and it gives https for voice for free), **not** `file://`. Tell me if this is
the scenario and I'll add a deploy step.

**Still needs your call:**

1. **Offline-safety is currently a myth.** The brief's airplane-mode test would show
   the palette working but the **page looking broken** — Tabler menu icons and Google
   Fonts come from CDNs today (already noted as a caveat in the project memory). If
   presentation-safety is the real goal, vendoring Tabler + the two font families is a
   separate small task worth doing at the same time. Recommend doing it.

2. **~25 MB model download** on first load. Fine over a local server (instant after the
   first run), but it means *rehearse on the actual presentation machine* — a cold
   cache on stage is the one thing that would visibly stall.

**Effort — v0: S–M** (a few days). Low technical risk; every piece is small and
self-contained.

## 8. Porting to the production app (QubesenseNextGenOPRSV2)

**Decision: demo ships at 41 pages; production is a later, separate project.** This section
exists so the demo doesn't paint us into a corner. Everything below was **verified in the
real codebase**, not assumed — it is the reason for several otherwise-arbitrary choices above.

### 8.1 The thing that kills "just index all the pages"

The production corpus is **not a fixed list**. Page access is per-user × per-client grants in
**`dbo.ClientSubMenuUser`** (`Emportal.WebDB/dbo/Tables/ClientSubMenuUser.sql:6-17`), enforced
only in `PageBase.OnInit` (`Old_App_Code/PageBase.cs:476-529`) by a URL string match. There are
no roles.

A static all-pages index would surface pages the signed-in user has no grant for — and that is
**not cosmetic**:

> **Verified: 162 of 257 `PageBase` pages set `RequiresReportAuthorization = false`** (89 set
> it `true`). On roughly two-thirds of pages the app **does not check page access at all**. An
> unfiltered palette would be the only thing between a user and a page they were never granted
> — a navigation feature quietly becoming an access-control hole.

**The fix is cheap, which is the good news.** `BindReportList()`
(`SiteV3.master.cs:300-322`) already caches `Session["clientMenu"]` / `Session["clientSubMenu"]`,
already filtered by client *and* per-user grants. Production emits the index **already filtered**
from session. The palette itself never filters and needs no permission code — so the demo's
"no permission model" is **not a shortcut to undo later**; it's the same code path with an
unfiltered input.

### 8.2 `pageId` is a fiction — why URLs are pre-built in the index

`1750` / `1455` are **`.aspx` filenames**, not DB keys. Nothing joins on them. The real
identifiers are `MenuId` + `SubMenuId` (IDENTITY PKs on `ClientMenu` / `ClientSubMenu`), and the
canonical URL is:

```
/Controls/OPRS/1750.aspx?MenuId=1108&SubMenuId=6905&ReportID=
```
*(verified: `SiteV3.master:398`, `SiteV3.master.cs:280`)*

Those query params are load-bearing — `SiteV3.master.cs:114-139` reads them for breadcrumbs.
**A URL cannot be derived from a pageId.** Hence §3.2: the index carries a finished `url` and
the widget only ever does `location.href = result.url`. This single decision is what reduces the
port to swapping the index producer.

### 8.3 What ports, and what doesn't

| Piece | Ports? |
|---|---|
| `command-palette.js` (ranking, Ctrl+K, keyboard nav, fallback) | ✅ unchanged — reads `window.__OPSFLO_PAGE_INDEX__`, agnostic to its origin |
| `generate_embeddings.js` | ✅ unchanged — consumes the same `pages.json` shape |
| Navbar markup | ✅ near-1:1 — the demo's navbar mirrors `SiteV3.master`; production already has the slot at `:124`, just `d-none` |
| The `pages.json` **producer** | ⚠️ **replaced** — Python `PAGES` dict → the 238 feature docs + a per-user session filter |
| Embedding text (§3.1) | ⚠️ **upgraded** — the `Documentation/Features/*.md` "AI Quick Reference" blocks are far richer than menu labels, and should mostly remove the need for hand-written `search` text |
| **v1 deep-linking** (§4) | ❌ **does not port** — demo-only theatre over pre-baked rows. Production pages use Kendo grids with real server-side filters + `GridStateController`; a real v1 there is per-page work and a separate design |

### 8.4 Production constraints to respect *now* (cheap now, expensive later)

- **Vanilla JS only.** `SiteV3.master` loads **no jQuery**; it's included per-page at
  inconsistent versions, and 8 SiteV3 pages have none. The demo plan is already vanilla — keep it.
- **One master ≠ every page.** `SiteV3.master` covers **140 of 262** pages.
  `SiteBlankV3.master` (63 pages) has **no navbar** — it's the popup/iframe shell, correctly
  out of scope. 48 more sit on 3 legacy shells.
- **A rollout switch already exists.** `Session["IsGlobalSearch"]` is already populated
  (`SiteV3.master.cs:268`) and toggleable from the admin UI at `Controls/OPRS/2865.aspx:97-98`
  ("Enable Global Search"). Its legacy consumer points at a **nonexistent `WebForm1.aspx/GetData`**
  and is dead code — the flag and its admin toggle are free to reuse.

### 8.5 The one production decision deliberately deferred (unchanged by the build)

**Where the query embedding runs.** In the demo, in-browser transformers.js is unambiguously
right. In production, a **~25 MB model per user** is a real cost that a demo doesn't have to
answer for. The alternative is a small server-side embed endpoint (ONNX Runtime in .NET, or a
hosted embeddings API) with corpus vectors in the DB.

This is exactly the call the "provider-agnostic, one function" abstraction (§1) exists to keep
open. **Do not decide it now** — decide it when production is greenlit, on real user-population
and network facts rather than demo convenience.

## 9. What the build changed about this plan

Written **after** implementing. Where §1–§8 and this section disagree, this section is
right. Five things only showed up once real code met the real browser.

### 9.1 Ranking is hybrid now, and both constants were measured

§3.4 specified pure cosine. Pure cosine scored **18/20** on the query set: it ranked
"who we work for" *third*, behind Account Managers Setup, even though that exact phrase
is in Customer Master's `search` text — mean pooling dilutes a phrase across a long
string. Adding a small lexical tiebreaker (`LEX_WEIGHT = 0.10`) takes it to **20/20**.

| lexW | top-1 | noise ceiling | weakest true positive |
|---|---|---|---|
| 0.00 | 18/20 | 0.195 | 0.212 |
| **0.10** | **20/20** | **0.195** | **0.312** |

The property that makes this safe: gibberish has no lexical overlap, so the weight
**cannot lift the noise ceiling** — it stays 0.195 at every weight and only ever rewards
true positives. That widened gap is what makes `MIN_SCORE = 0.25` sit comfortably
between noise and the weakest real hit. Semantic still dominates (cosine spans
~0.2–0.63); 0.10 is a tiebreaker, not keyword search in disguise. **Both numbers came
from measurement — re-measure if the corpus changes, don't hand-tune.**

### 9.2 `transformers.min.js` is an ES module, so there is no `<script>` global

§3.3 said to add `<script src=".../transformers.min.js">`. That silently does nothing —
the dist is an ES module (`export { pipeline, … }`), not a UMD bundle, so it exposes no
global. The widget instead **lazy-loads it with a dynamic `import()`** from a classic
script. This is strictly better: the `file://` fallback then happens for free, because
that's exactly where module loading is CORS-blocked.

> **The bug this nearly shipped as:** `import()` treats a specifier not starting with
> `./`, `../` or `/` as a **bare module specifier**. `SITE_ROOT` is `"../"` inside
> `pages/` (fine) but `""` at the root — so `import("assets/…")` threw only on
> `index.html` and `launcher.html`, i.e. **the two pages the demo opens on**, while the
> 41 inner pages worked. It degraded to keyword mode silently, with no error banner. The
> fix is one `new URL(…, document.baseURI)`. Nothing but a real browser would have
> caught this.

### 9.3 Do not use Frest's `--bs-tertiary-bg` / `--bs-secondary-bg`

Frest overloads these with non-Bootstrap semantics: **in dark mode
`--bs-tertiary-bg` is `rgba(255,255,255,.75)` — near-opaque white.** Using them as
backgrounds paints light-grey rows on the dark panel. `command-palette.css` defines its
own `--cmdp-*` tokens under `.light-style` / `.dark-style` instead. Safe to use directly:
`--bs-primary`, `--bs-body-bg`, `--bs-body-color`, `--bs-border-color`,
`--bs-heading-color`, `--bs-secondary-color`.

Two related traps, both fixed and commented in the CSS:
- `core-dark.css` sets `input.form-control { background-color:#3b3b3b !important }`. An
  author `!important` beats any specificity, so the palette's transparent input needs
  `!important` back. It is load-bearing, not laziness.
- The menu is positioned by hand (`right:0`). We toggle `.show` directly rather than use
  `data-bs-toggle="dropdown"` (Bootstrap's dropdown steals focus and closes on clicks
  inside), which means Popper never runs and `.dropdown-menu-end` never applies — a
  420px panel would overflow the viewport and clip its own footer.

### 9.4 Sizes and timings, measured

- Vendored total **61 MB** on disk, but a browser downloads **~33 MB** once: it fetches
  one WASM variant, not all four (`ort-wasm-simd.wasm` 9.5 MB + `model_quantized.onnx`
  21.9 MB + `tokenizer.json` 0.7 MB + `transformers.min.js` 0.9 MB). §6 said ~25 MB.
- `page-index.js` is **155 KB** for 41 pages × 384 dims.
- Model ready **~2.1 s** after page load on a warm cache; queries are then instant.
- §6 #2 stands and is the real risk: **rehearse on the presentation machine**, because
  a cold cache is the one thing that would visibly stall.

### 9.5 v1 landed as designed, with one simplification

The deep-link highlight lives in `command-palette.js` (which is already on every page),
not as a separate inline script in `content_for()` as §4 suggested. That avoids
`str.format()` brace-escaping entirely and keeps the feature in one file. Behaviour is
unchanged: `?status=Overdue` highlights matching rows, dims the rest, shows a count with
a Clear button. **Still demo-only — it does not port (§8.3).**

## 10. Second pass — suggestions + dual-mode input (BUILT)

> Naming: §§3–5 are the original feature tiers (v0 semantic, v1 deep-link, v2 voice). This
> section is a later pass over all of them — not a fourth tier.

One input now serves **both** "take me to Payroll / 1676" and "who is late", and the empty
state is a live, clickable surface. 40/40 automated checks (28 new + 12 regression).

### 10.1 Two lanes, merged by rank — never by score

A **name lane** (deterministic, synchronous, model-free: id/title/section tier ladder) is
merged with the existing **meaning lane**. `mergeLanes()` layers names on top **by
position**; no arithmetic ever crosses the two.

> **This is the design, and the most likely thing to be "simplified" into a bug.** Name
> tiers (0.40–1.00) and cosine (~0.20–0.63) are different scales. `MIN_SCORE = 0.25` is not
> a general confidence threshold — it is a measured property of `dot + 0.10*lexical`
> (§9.1). Adding a tier to a cosine, or filtering name hits through `MIN_SCORE`, destroys
> the only thing those numbers mean. Because the semantic lane is untouched, **v2 needed no
> re-measurement** — and §9.1's 20/20 still holds (re-verified).

Guards that keep the name lane from hijacking real queries: `PIN_TIER = 0.80` (a substring
is not an identification — `report` matches 6 titles and pins *nothing*) and
`AMBIG_MAX = 6` (`16` id-prefixes 10 pages → pins nothing). `NAME_PIN_MAX = 3` structurally
reserves 2 of 5 slots for meaning.

**`id` is deliberately NOT in the embedded text nor in `_hay`.** Mean pooling would shred
`1676` into contentless subwords and dilute every other token in the vector; touching `_hay`
would silently move the measured `LEX_WEIGHT`. A string compare gets this right 100% of the
time — an embedding cannot promise that.

### 10.2 Three real bugs this surfaced

1. **`keywordScore` matched substrings, not words.** `"who's late"` returned **Vehicle /
   Truck Master** — "late" is inside *"plate"*, *"template"* and *"related"*, giving a 4-way
   tie at exactly `MIN_SCORE` broken by id order, with the right answer 4th. Live under
   `file://` and in the ~2.1s cold-model window — i.e. **the demo's own headline example was
   wrong on the two paths nobody tests.** Fixed with prefix-anchored matching (`\blate`
   matches "late" not "plate"; `\bcertification` still matches "certifications"). Fallback-only
   ⇒ no re-measurement debt.
2. **The palette closed on its own clicks.** The outside-click handler used
   `root.contains(e.target)`, but rendering replaces `innerHTML` during the microtask
   checkpoint *between listeners*, **detaching the clicked node mid-dispatch**. A detached
   node is contained by nothing → `close()`. Only visible once the empty state became
   clickable. Fixed with `composedPath()`, which is computed at dispatch and survives detachment.
3. **`preview()` must `++seq`.** The stale-response guard didn't just miss the instant lane —
   it *passed* it: `run("payr")` takes `mine = seq = 1` and starts a ~40ms embed; the user
   types `o`; `preview("payro")` renders; the stale run resolves and finds `seq` **still 1**
   (payro's debounce hasn't fired) → stale results clobber the fresh render. `seq` means "a
   newer query state exists", and `preview()` creates one.

### 10.3 The example gate

`generate_embeddings.js` now **fails the build** unless every `Try asking` example is top-1
in **both** semantic and keyword mode. Semantic-only would have missed bug #1 entirely. This
turns §7.4 from a manual checklist into CI.

> The gate **mirrors** `keywordScore`/`lexical` from `command-palette.js`. Keep them in sync
> — a divergence means the gate stops testing what ships. (Caught during the build: the
> mirror had the fix before the widget did, so the gate went quiet.)

### 10.4 Suggestions

Empty state and zero-results share **one code path**: `Try asking` (3 build-verified NL
examples) → `Recent` (≤3) → `Jump to` (curated defaults top the nav zone back to 3). Always
6 rows, so a fresh machine, a used one, and one where `localStorage` threw all look identical
— **the fallback path and the failure path are the same path.**

**Recents store page IDs, never query text**, hydrated from `INDEX` (`opsflo-recents:<userKey>`,
LRU, max 5). Presentation-safe (titles come from the curated `PAGES` dict, so a leaked list
reads as intentional); portable (a stored url would strand broken links across the port); and
**permission-safe in production for free** — an id the user has no grant for simply fails to
hydrate and drops out, so `.filter(Boolean)` *is* the check (§8.1).

`.cmdp-results` max-height went 320 → **380px**: the fullest state is 3 headers + 6 rows
≈ 327px. At 320 it scrolled *only once a recent existed*, which a fresh demo machine never
shows — it would have appeared for the first time mid-presentation.

### 10.5 A/B — pick one and delete the other

`'badge'` (default) vs `'grouped'`. Additive: grouped = badge + `PAGES`/`BEST MATCHES` headers.
```
__opsfloPaletteStyle('grouped')   // console, persists, re-renders live
?palette=grouped                  // URL param, for a side-by-side
__opsfloClearRecents()            // run-book: wipe rehearsal residue
```
**Scaffolding, not a feature — delete the loser once chosen.** Name hits show the page ID where
AI hits show a similarity score: `Math.round(1.00*100)` = "100" beside a semantic "34" is
meaningless *and* undercuts what the score exists to prove (`command-palette.css`: "evidence
that this is ranked semantic matching and not a substring filter") — a name hit **is** a
substring filter.
